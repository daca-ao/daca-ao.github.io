远程过程调用 RPC

Posted on 2021-07-31

全称 Remote Procedure Call Protocol，远程过程调用协议

应用背景：

  • 以前只需要一个服务器，将程序全部打包好便可提供服务
  • 随着流量增大，常规的垂直应用架构已无法应对

完整框架

功能组成：
客户端（Client）：服务调用方
客户端存根（Client stub）

  • 存放服务方地址信息
  • 将客户端的请求参数数据信息打包（marshalling，或序列化）成网络消息，再通过网络传输发送到服务端
  • stub 为一辅助对象，不执行任何实际功能
    服务端存根（Server stub）
  • 接收客户端发送过来的请求信息，并进行解包（unmarshalling）
  • 再调用本地服务进行处理
    服务端（Server）：服务真正提供者
    网络服务：底层传输，可为 TCP 或 HTTP

一次 RPC 的过程：

  1. 客户端通过本地调用，执行需要调用的语句，并传递参数
  2. 客户端 stub 接收到调用请求后，将方法、入餐等信息序列化（组装、marshalling）成能够进行网络传输的消息体
  3. 客户端 stub 找到远程服务地址，发送网络消息
  4. 服务器 stub 收到信息后，解析（unmarshalling，或反序列化）取得方法名和参数
  5. 服务器 stub 根据解析的结果，调用本地服务执行远程过程（服务）
  6. 服务端执行过程完毕，将结果返回服务器 stub
  7. 服务器 stub 将处理结果序列化
  8. 服务器 stub 通过网络服务发送结果至客户端
  9. 客户端 stub 接收到结果，对结果进行解码
  10. 客户端接收到调用语句返回的结果数据

实现 RPC 的技术点：

  1. 服务寻址
  • 使用 Call ID 映射
    • 原因：
      • 函数体由函数指针指定，但
      • 远程调用中两个进程的地址空间完全不同，函数指针无效
    • RPC 中所有函数必须有自己的 ID，该 ID 在所有进程中唯一确定
    • 客户端做远程调用时必须附上 ID
  • 另：客户端和服务端分别维护 “函数 <—> Call ID” 对应表
    • 客户端执行远程调用：查表，找出对应 ID，传给服务端
    • 服务端通过查表，确定客户端需要调用的函数，执行
  • 实现方式：服务注册中心
  • 实现案例：RMI
  1. 数据流的序列化和反序列化
  • 本地调用：将参数压入栈，函数自己去栈里读
  • 远程调用：客户端和服务端不能通过内存传递参数
  1. 网络传输
  • 客户端和服务端通过网络连接，所有数据需要通过网络传输
  • 需要一个网络传输层传递 Call ID + 序列化的参数字节流（去），以及序列化的调用结果（来）
  • 协议不限
    • 大部分 RPC 框架使用 TCP 协议（最常见）
    • gRPC 使用 HTTP2

具体的 RPC 实现：

  • Call ID：可直接使用函数字符串，也可使用整数 ID；映射表一般为哈希表
  • 序列化与反序列化：可自己写，也可使用 Protobuf 或 FlatBuffers 等
  • 网络传输库：可自己写 Socket，也可用 Asio, ZeroMQ, Netty 等

使用场景（大型网站，内部子系统较多、接口非常多的系统）：

  • 长链接：不必每次通信都像 HTTP 一样三次握手，减少网络开销
  • 注册发布机制：监控管理、发布、下线接口、动态扩展
  • 安全性：没有暴露资源操作
  • 微服务支持

RPC v.s. RESTful

  1. 面对对象不同
  • RPC 更侧重于动作，REST 的主体是资源
  • REST 是面向资源的设计架构，但系统中很多对象不能抽象成资源，如登录，修改密码等
  • RPC 可通过动作去操作资源
    • 操作全面性：RPC > REST
  1. 传输效率不同
  • RPC 使用自定义 TCP 协议，请求报文体积更小，效率更高
  • HTTP2 协议也可很好地减少报文体积
  1. 复杂度不同
  • RPC 实现复杂，流程繁琐
  • REST 调用和测试都很方便
  1. 灵活性
  • HTTP 相对更规范，更标准，更通用，不限制于语言
  • RPC 可实现跨语言调用，但整体灵活性不如 REST

RPC or HTTP?

  • HTTP 接口在接口不多、系统与系统交互较少时，解决信息孤岛初期常使用的通信手段
    • 优点：简单、直接、开发方便
  • 如系统内部子系统较多，接口多时，RPC 优势更大
    • 长连接，不必每一次通信都像 HTTP 一样进行三次握手，减少网络开销
    • RPC 的注册中心的监控管理功能丰富，可发布、动态扩展等
    • 安全
