数据库事务

Posted on 2021-07-25

事务（Transaction）：访问并可能更新数据库中各种数据项的一个程序执行单元

  • 一系列的数据库操作，管理 INSERT, UPDATE, DELETE 语句
  • 须具备 ACID
  • 局部事务
    • 特定于一个单一的事务资源，如一个 JDBC 连接
    • APP 组件与资源位于同一单元
  • 全局事务
    • 跨多个事务资源的事务，如在一个分布式系统中的事务

可以是：

  • 一条或一组 SQL 语句
  • 整个程序
  1. 原子性（Atomicity）
  • 一个事务中的操作，要么全部完成，要么全部不完成（不做）
  • 不会结束在某个中间环节
  • 执行若发生错误：回滚（rollback）到事务开始前的状态，就如事务从未被执行
  1. 一致性（Consistency）
  • 事务开始前和开始后，数据库完整性没有被破坏
  • 表示写入的资料必须完全符合所有的预设规则
  • 包含资料的精确度、串联性以及后续数据库可以自发性完成预定的工作
  1. 隔离性（Isolation)
  • 数据库允许多个并发事务同时对其数据进行读写和修改的能力
  • 并发事务之间不能互相干扰
  • 可防止多个事务并发执行时，由于交叉执行而导致数据的不一致
  1. 持续性（Durability）
  • 事务处理结束后，对数据的修改是永久的，即使系统故障也不会丢失

并发会破坏数据库事务的隔离性：

并发下的问题 描述
脏写 事务回滚了其它事务对数据项的已提交修改
例：
     1. 事务 1 读取 A = 10（未提交）
     2. 随后事务 2 读取 A = 10，写回 A = 30，并提交（脏写）
     3. 随后事务 1 操作需要回滚，A 被回滚到 10，而不是已经提交的 30
丢失更新 事务覆盖了其它事务对数据的已提交修改
例：
     1. 事务 2 读取 A = 10
     2. 事务 1 读取 A = 10
     3. 事务 2 执行 A += 10，写回 A = 20，并提交
     4. 事务 1 执行 A -= 10，因其读取到的 A 的值是 10，执行后 A = 0
     5. 事务 1 提交，导致在其之前的事务 2 的提交被覆盖（丢失事务 2 的更新）
脏读 Dirty Read 一个事务读取了另一个未提交的事务中的数据
不可重复读 Non-repeatable Read 对于数据库中某个数据，在同一个事务范围内多次查询却返回不同数据值
因为在查询间隔，该数据被另一个事务修改并提交了
例：
     1. 事务 1 读取 A = 10
     2. 事务 2 读取 A = 10，执行 A *= 2，写回 A = 20，并提交
     3. 事务 1 读取 A = 20，与之前读取的 A = 10（已是不可重复的读取）不一致
幻读（多行的不可重复读）Phantom 一个事务多次查询之间的数据的条数不一致
针对一批数据整体出现的问题
例：
     1. 事务 1 读取 A < 5 的数据有 (1, 2, 3)
     2. 事务 2 写入 A = 4，并提交
     3. 事务 1 读取 A < 5 的数据有 (1, 2, 3, 4)，与之前读取的不一致
脏读示意 Session A Session B
1 begin;
2 begin;
3 update city set name=’’ where id = 1;
4 select * from city where id = 1;
5 commit;
6 rollback;

脏读 v.s. 不可重复读

  • 脏读读取的是事务未提交的脏数据
  • 不可重复读读取的是事务已经提交的数据

幻读 v.s. 不可重复读

  • 不可重复读针对确定的某一行数据
  • 幻读针对不确定的多行数据（Result Set）

事务隔离级别

逐级提高：

  • 读未提交（Read Uncommitted）：所有事务都可看到其它事务修改过但未提交的内容数据
    • 可能发生脏读、不可重复读和幻读问题
    • 没有解决任何并发问题，不常用
  • 读提交（Read Committed）：一个事务只能读取其它事务已经提交的内容数据
    • 如：事务 B 只能在事务 A 修改过并且提交之后，才能读取到事务 B 修改的数据
    • 解决了脏读问题，但没有解决不可重复读和幻读，也不常用
    • Oracle 的默认级别
  • 可重复读（Repeatable Read）：保证一个事务之间的多个实例在并发下能读取同一数据
    • 对同一字段的多次读取结果是一样的，除非数据被自身事务修改
    • MySQL InnoDB 的默认级别
  • 串行化（Serializable）：事务之间只能顺序执行，使之没有任何冲突
    • 最高级别
    • 通过加锁实现（读锁和写锁）

级别越高，越能保证数据完整性和一致性，对并发效率也越低。

对象状态 脏读 不可重复读 幻读
读未提交 可能 可能 可能
读已提交 不会 可能 可能
可重复读 不会 不会 可能
串行化 不会 不会 不会
