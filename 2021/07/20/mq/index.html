消息队列与其实现

Posted on 2021-07-20

消息队列是中间件的一种，是分布式系统中非常重要的组件。

消息队列（Message Queue, MQ）的通用场景可简单描述为：

当不需要立即获得结果，但是并发量又需要进行控制的时候。

消息队列（Message Queue, MQ）

  • 中间件的一种，分布式系统中重要的组件
  • 应用程序之间的通信方法
    • 应用场景：解耦、异步消息、流量削锋等问题
    • 生产者-消费者的典型代表
  • 实现高性能、高可用、可伸缩和最终一致性架构
  • 是大型分布式系统不可缺少的中间件
    队列的使用除去了接收和发送应用程序同时执行的要求

生产环境使用较多的消息队列：ActiveMQ, RabbitMQ, ZeroMQ, Kafka, MetalMQ, RocketMQ 等

选型依据：

MQ 的消息传播模式：
推（push）模式：服务器主动将信息送到客户端
优点：

  • 对用户要求低，方便用户获取需要的信息
  • 及时性好，服务器端即时向客户端推送更新的动态信息
    缺点：
  • 不能确保发送成功
    • push 采用广播方式，只有服务器端和客户端在同一频道上，推模式才有效，用户才能接收到信息
  • 没有信息状态跟踪
    • push 采用开环控制技术，一个消息推送后的状态，如客户端是否接收等无从得知，针对性较差
    • 推送的消息可能并不满足客户端的个性化需求

拉（pull）模式：客户端主动发起获取信息

  • 如没有消费者在监听队列，消息将保留在队列中，直至消费者连接到队列，取到信息为止
  • 该模式下消息不实自动推送给消费者的，而是由消费者从队列中请求获得
    优点：
  • 针对性强，满足客户端个性化需求
  • 信息传输量较小，网络中传输的只是客户端的请求和服务器端对请求的响应
  • 服务器端任务轻，只是被动接受查询，对客户端的查询请求作出响应
    缺点：
  • 实时性差，针对服务器端实时更新的信息，客户端难以获取实时信息
  • 对于客户端用户的要求较高，需要对服务器端具有一定的了解

优点：

  • 实现异步、解耦、流量削峰填谷
  • 可解决以下特殊业务场景的问题

缺点

消息延迟推送应用：

  • 淘宝七天自动确认收货
    • 签收后，物流系统在七天后延时发送一个消息给支付系统，通知支付系统七天后将款项打给商家
  • 12306 购票支付确认界面
    • 选好票点击确定跳转的页面往往会有倒计时
    • 如 30 分钟内订单不确认将自动取消订单
    • 如 30 分钟内完成订单，则根据逻辑代码确定忽略接收到的延时信息

缺点：

  • 系统可用性：万一 MQ 崩溃了就完了
  • 提高了系统复杂性，需要考虑一些消息队列的常见问题和解决方案
  • 一致性问题
    • 多个消费者中其中一个消费失败，会导致数据不一致
    • 会有消息重复消费、消息积压、消息顺序和消息事务的问题

重复消费问题：通常由网络原因造成
正常情况：

  1. 消息被成功消费后，消费者发送成功标志给 MQ
  2. MQ 收到标志后，表示消息被成功消费，不会再将消息发送给其他消费者
    问题：
  3. 成功标志因网络问题在传输过程丢失
  4. MQ 没接收到该标志，认为该消息没有被成功消费
  5. MQ 再次发送该消息给其他消费者，造成重复消费
    解决问题：保证消费端幂等性
    主要原理：去重
    方法1：
  • 每个消息生成一个 id
  • 在第三方介质（如 Redis set）中准备 <消息 id, 消息>
  • 开始消费前，在 Redis 中查询有无该记录即可
    方法2：针对数据库插入操作
  • 以消息作为唯一主键，重复消费时会导致主键冲突
    • 下游业务接收到 message 后，执行业务并保存 message ID

消息丢失问题
无非就是三种情况：

  1. 生产者弄丢数据
  2. 消息队列弄丢数据
  3. 消费者弄丢数据

解决方法：

  1. 主流 MQ 都有确认机制或事务机制，可保证生产者将消息送达 MQ。如：RabbitMQ 就有事务模式和 confirm 模式
  2. 只要开启 MQ 的持久化磁盘配置就能解决 MQ 弄丢数据的问题
  3. 消费者丢数据一般是因为采用了自动确认消息模式，MQ 收到确认消息后会删除消息，如果消费者发生异常，消息就没了
    1. 改用手动确认就能解决

消息顺序问题
场景：电商平台的下单操作，下单后先减库存，然后生成订单

  • 首先生产者需要保证入队的顺序
  • 发送消息时通过 hash 取模
    • 确保同一操作的消息发送到同一个队列中
    • 一般的 MQ 中同一个队列都能保证 FIFO，由此确保出队时是顺序的
  • 如多个消费者同时消费一个队列，一样可能出现顺序错乱的情况
