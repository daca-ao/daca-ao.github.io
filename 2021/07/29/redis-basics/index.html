Redis 入门

Posted on 2021-07-29

在非关系型数据库（NoSQL）中，Redis 绝对是 C 位的存在。今天就开一个帖来专门唠唠 Redis。

简介

Redis，全称 REmote DIctionary Server。官方解释是“BSD 开源许可的，使用 ANSI C 语言开发的，部署于内存的数据结构（in-memory data structure store）”。
Redis 不仅能用来做内存数据库，还可以做缓存和消息代理。

Redis 支持多种数据结构：

  • 字符串 string
  • 列表 list
  • hash
  • 集合 set
  • 带范围查询的有序集合 sorted set(zset)
  • 位图 bitmap
  • hyperloglog

前五种为基本数据类型，后两种为高级对象。

Redis 具有内置的复制，Lua 脚本，LRU 淘汰策略，事务和不同级别的磁盘持久性，并通过 Redis Sentinel 和 Redis Cluster 的自动分区提供高可用性。

Redis 所有的单个操作都是原子性的，多个操作支持事务。


在详细介绍 Redis 的数据类型之前，先来看看 Redis 数据存储的具体模型：

内存模型

Redis 作为 NoSQL 类型的内存数据库，在内存中主要以键值对（key-value）的方式存储内容。

内存统计：

其中：

used_memory

  • Redis 分配器分配的内存总量（Byte）
  • 包括使用的虚拟内存（swap）

used_memory_human

  • used_memory 更友好的显示方式

used_memory_rss

  • Redis 进程占据操作系统的内存（Byte），与 top 和 ps 命令看到的值一致
  • 包括：
    • Redis 分配器分配的内存
    • 进程运行本身需要的内存
    • 内存碎片
    • ……
  • 不包括虚拟内存

used_memory 与 used_memory_rss 比较：

  • 前者为从 Redis 得到的量
  • 后者从操作系统角度得到的量
  • 两者不同的原因：
    1. 内存碎片和 Redis 进程运行需要占用内存：前者可能比后者小
    2. 存在虚拟内存：前者可能比后者大

mem_fragmentation_ratio

  • 内存碎片比率，used_memory_rss / used_memory
  • 实际应用中 Redis 数据量比较大：进程运行占用内存与 Redis 数据量和内存碎片相比会小很多
  • 一般大于 1，值越大内存碎片比例越大
  • 小于 1，说明 Redis 使用了虚拟内存：要及时排查
  • 1.03 为比较健康的状态

mem_allocator

  • Redis 使用的内存分配器，编译时指定
  • 可以是 libc, jemalloc 或 tcmalloc，默认 jemalloc

jemalloc 将内存空间划分为小、大、巨大三个范围，每个范围内划分许多小的内存块单位。

存储二进制数据

内存占用主要有以下部分：

  1. 数据
  • 最主要的部分，其占用内存统计在 used_memory 中
  • 每种类型可能有 2 种或更多的内部编码实现
  • 存储数据对象时：并不是直接将数据扔进内存，而是对对象进行各种包装：如 redisObject、SDS 等
  1. 进程本身运行需要的内存
  • 代码、常量池等
  • 大多数场景中，这部分的内存占用可忽略
  • 不由 jemalloc 分配，因此不会统计在 used_memory 中
  • 除主进程外，Redis 创建的子进程也会占用内存：执行 AOF、RDB 重写时创建的子进程
    • 这部分内存不属于 Redis 进程，不会统计在 used_memory 和 used_memory_rss 中
  1. 缓冲内存，包括：
  • 客户端缓冲区：存储客户端连接的输入输出缓冲
  • 复制积压缓冲区：用于部分复制功能
  • AOF 缓冲区：在进行 AOF 重写时，保存最近的写入命令
  • …
  1. 内存碎片
  • Redis 分配、回收物理内存过程中产生的
    • 如：对数据的更改频繁，且数据之间的大小相差很大
    • 可能导致 Redis 释放的空间在物理内存中并没有释放，但 Redis 又无法有效利用：形成内存碎片
  • 不会统计在 used_memory 中
  • 其产生与对数据进行的操作、数据的特点等有关
  • 与使用的内存分配器有关：如内存分配器设计合理，可尽可能减少内存碎片产生
  • 如内存碎片很大：安全重启减小内存碎片
    • 重启后 Redis 重新从备份文件中读取数据，在内存中进行重排

数据存储的细节
例：

  • 对 Redis 每个键值对都有一个 dictEntry，存储指向 key 和 val 的指针；next 指向下一个 dictEntry
  • key 为 “hello”：并不是以字符串存储，而是存储在 SDS 结构中
  • val 为 “world”：不以字符串或 SDS 结构存储，而是存储在 redisObject 中
    • type：指明 val 对象的类型
    • ptr：指向对象所在的地址
    • 字符串对象 “world” 仍存储于 SDS
  • 内存分配器分配内存
    • 如：dictEntry 有三个指针，占用 24 字节：分配 32 字节

lru：通过对比 lru 时间与当前时间，可计算某个对象的空转时间（object idletime）

  • 除了通过 object idletime 打印外，还与 Redis 内存回收有关系
  • 如 Redis 打开了 maxmemory 选项，且内存回收算法选择 volatile-lru 或 allkeys-lru
    • 当 Redis 内存占用超过 maxmemory 指定值时，Redis 会优先选择空转时间最长的对象释放

refcount

  • 主要用于对象引用计数和内存回收
    • 创建新对象：refcount=1
    • 有新程序使用该对象：refcount + 1
    • 对象不再被一个新程序使用：refcount - 1
    • refcount = 0：对象占用内存会被释放
  • 被多次使用的对象称为共享对象（refcount>1）
    • 某些对象重复出现时，新程序不会创建新对象，而是仍然使用原来的对象
    • 节省内存
    • 仅支持整数值的字符串对象
  • 共享对象具体实现
    • 只支持整数值的字符串对象：对内存和 CPU（时间）的平衡
      • 共享对象虽降低内存消耗，但判断两个对象是否相等需消耗额外时间
      • 整数值判断操作复杂度为 O(1)；普通字符串为 O(n)；列表等为 O(n2)
    • 五种类型都可以使用共享对象
    • Redis 服务器初始化时会创建 10000 个字符串对象：0 - 9999
      • Redis 需要使用值为其中的字符串对象时可直接使用

数据类型

Redis 是基于键值对的数据库，其中“值”的类型有：

String

  • Redis 最基本的数据类型，长度不超过 512 MB
  • 二进制安全，可以包含任何数据，如 jpg 图像或者序列化的对象

Redis String 有以下编码方式：

  1. int：8 个字节的长整型
    1. 字符串值为整型时，该值使用 long 整型表示
  2. embstr：<= 39 字节的字符串
  3. raw：大于 39 字节的字符串

编码说明：

  • embstr 与 raw 都使用 RedisObject 和 SDS 保存数据
  • embstr 的使用只分配一次内存空间（RedisObject 和 SDS 连续）
    • 好处：创建时少分配一次空间，删除时少释放一次空间；对象所有数据连在一起，寻找方便
    • 坏处：字符串长度增加需要重新分配内存时，整个 RedisObject 和 SDS 都需要重新分配空间
    • 因此 embstr 实现为只读
  • raw 需要分别为 RedisObject 和 SDS 分别分配内存空间

String 编码转换：

  • 当 int 数据不再是整数，或大小超过 long 范围：自动转为 raw
  • 因 embstr 的实现是只读：修改时先转化为 raw 再进行修改

Hash

  • 键值对（key-value）集合
  • 本质为 String 类型的 field 和 value 的映射表
  • 指令以 h 开头

List

  • 简单的 String 列表，按照插入顺序排序
  • 可添加至列表头（左）或尾（右）
  • 指令以 l 开头

Set

  • 元素为 String 的无序不重复集合，通过哈希表实现
  • 指令以 s 开头

zset（Sorted Set）

  • String 有序不重复集合，与 Set 类似
  • 每个元素关联一个 double 类型的分数，Redis 通过分数从小到大排序
  • 指令以 z 开头

为啥每种对象类型都有至少两种编码？

  1. 接口与实现分离，当需要增加或改变内部编码时，用户使用不受影响
  2. 可根据不同应用场景切换内部编码，提高效率

优点

Redis 是单进程单线程的模型，对于数据的操作完全基于内存，因此 CPU 不是瓶颈；而且单线程容易实现。
为何 Redis 是单线程模型，处理起来还仍然这么快？

  • 数据结构简单，对数据的操作也简单
  • 数据存储类似于 HashMap：查找和操作的时间复杂度是 O(1)
  • 采用单线程，避免不必要的上下文切换和竞争条件
  • 使用非阻塞 IO
