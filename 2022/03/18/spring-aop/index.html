Spring AOP 概述

Posted on 2022-03-18

同样，AOP 不仅仅是针对 Spring 才能应用的逻辑。

AOP 概述

我们知道，OOP 可以提供复杂而纵深的继承体系，但是对于一些非继承关系的横向加强功能并不容易实现。

比如，当很多其它的类的方法都调用某一个类的成员方法时，需要这些类都持有该类的对象，且这些类的所有对象都需要显式地依赖该对象。

举个例子：

A 和 B 两个类中的 doSth() 部分代码几乎是完全重复的，而且与其业务并不相关联。
有的时候虽然这种代码并不多，但是业务类膨胀了之后，万一都需要用到这一段逻辑，反复添加这种与业务不相关的代码也是一件比较烦的事。

能否用某种方式实现同样功能，而不需要开发者写重复的代码逻辑？

由此我们提出了 Aspect-Oriented Programming，面向切面编程，是一种对 OOP 纵向结构丰富而横向结构缺失的补充。

AOP 实现基础

  1. 利用编译器将这些功能代码在编译阶段“织入”源代码
    1. 运行时将代码切入到类的指定方法、指定位置
    2. 需要特殊的编译器
  2. 利用动态代理在运行时将这些功能代码切入

Java 动态代理

  1. JDK 代理
  • 通过反射接收被代理的类
  • 被代理的目标类需实现一个接口（业务相关的接口），没有在该接口定义的方法不能被代理
  • Spring AOP 默认采用此方式进行代理
  • 只有被代理的类没有实现接口时：才会被切换到 CGLib 代理
  1. CGLib（Code Generation Library）代理
  • 需要 CGLib 支持
  • 相当于运行时动态创建一个被代理的类的非继承子类（组合）
  • 也称为子类代理
  • final 类无法使用 CGLib 作为动态代理

AOP 相关概念

  1. 增强 Advice
  • 切面（Aspect）当中的方法
  • 声明增强方法在目标（Target）业务层的执行位置
  • 定义了 When 和 What，即 aspect 的切入时机，以及拦截后进行哪些行为
    • 如上例中的 TheOne.doSth()
  • 针对一个方法（What），合理的切入时机（When）包括：
    • @Before: 前置通知，方法调用前
    • @After: 后置通知，方法执行后，不考虑返回
    • @AfterReturning: 返回通知，方法返回后
    • @AfterThrowing: 异常通知，方法抛出异常后
    • @Around: 环绕通知，方法前以及方法后
  1. 切入点 Pointcut
  • 定义了 Where，即匹配那些切入的方法，确定在哪些类，哪些方法上切入
    • 归纳定义相应的连接点 Join Point，确定具体拦截的某个业务点
    • 即：一组已经被增强的2. 连接点
  • 允许具体的方法名、类名、正则匹配等
  1. 连接点 Join Point
  • 程序（业务流）中的概念
  • 可理解为增强（Advice）定义的上述切入时机 When
  • 即对哪些方法进行拦截，拦截后怎么处理
  1. 切面 Aspect
  • 拦截器类，包含增强（Advice）和切入点（Pointcut）的定义
  • 增强 + 切入点
  • 在 advice 与 pointcut 配合下，切面就成型了
    • 程序就知道切入的 When、Where 和 What
    • 什么时机，什么地方做什么增强
  1. 引入 Introduction
  • 上述切面（Aspect）只是针对指定方向的增强
  • 而引入可以针对类进行增强，为其动态添加新的方法或域

  1. 织入 Weaving
  • 参照 pointcut 的定义取得相应的 join points
  • 将相应的 advice 加入到 pointcut 指定的 join points，并创造出代理对象的过程
  • 由代理完成
  1. 目标对象 Target
  • 被一个或多个切面增强的对象，因此也被称为 被增强对象
  • 即：上述箭头的终点
  • 永远是一个被代理对象

常见应用：

  • 权限控制（鉴权）
  • 日志
  • 参数校验
  • 缓存、调试、性能检测
  • 一定程度的事务处理
  • …

AOP 思想中，功能分为：

  1. 核心业务（核心关注点），如登陆、增加数据、删除数据
  2. 周边功能（横切关注点），如性能统计、日志、事务管理等
  • 周边功能在 AOP 中被定义为切面
  • 核心业务功能与切面功能分别独立开发，然后“编织”到一起，就是 AOP
    目的：
  • 将与业务无关，却为业务所共同调用的逻辑或责任（事务处理、日志管理、鉴权、……）封装起来
  • 集中处理某一关注点/横切逻辑
  • 可以很方便地添加/删除关注点
  • 便于减少系统重复代码，降低模块间耦合度，利于可拓展性和可维护性

Spring AOP

Spring AOP 使用纯 Java 实现，不需要专门的编译过程和类加载器

  • 运行时通过代理方式向目标类织入增强代码
  • Spring 2.0+：引入对 AspectJ 的支持

实现原理：代理机制

  • 接口 + 实现类：采用 JDK 动态代理 Proxy
  • 实现类：使用 CGLib 字节码增强

通过 AnnotationAwareAspectJAutoProxyCreator 实现

  • 为一个 BeanPostProcessor
  • 父类为 AbstractAutoProxyCreator
  • 如给一个类中某个方法设置了切面：这个类最终需要生成一个代理对象，再放进 bean 缓存池中
