Java Map 接口

Posted on 2022-05-06 Views:

Java 集合类“二王”中的另一个，就是 Map 接口。

Map，意为映射表，表的元素是一个又一个的 Entry 对象。

每个 Entry 对象为一键值对（Key-Value pair），Key 与 Value 可以是任意的 Object。

由于要根据 Key 来区分不同的 Entry：Key 不能重复，Value 可以重复（包括 null）。

Map 不继承 Collection 接口：Map 不是集合，如继承自 Collection：键值对何去何从？作为一组对象存放于 collection？这样子会加大映射表的负担，增加操作的复杂度。

接口方法

JDK 为映射表的 Key 和 Value 规定了两个范型：<K, V>。

在这里，我们需要说一下 hashCode() 和 equals() 这两个重要的方法。


hashCode() & equals()

如果两个对象相等（equals() 返回 true），则散列值 hash code 一定相同；如两个对象 hash code 值相等，对象则不一定相等。

  • hash code 相等即键值对的 hash value 相等；
  • hash value 相等并不一定能得出键值对相等，不同的键值对可能会得出一样的散列值，这就是哈希冲突，又称散列冲突。

equals()，表示某个类逻辑上的相等，不同于 == 判断内存地址是否相等。

有需要的话，类需要重写 equals() 实现该类对象的逻辑比较判断。

现在，假设有某一个类：

如果在散列表数据结构（HashSet, HashTable, HashMap 等）中用到该类，hashCode() 与 equals() 有关系，否则两者没关系。
该类作为散列表的 key 的话，必须重写 hashCode()。

如果有必要重写 hashCode() 或 equals()，最好两个一起重写；如果重写了 equals()，也应重写 hashCode() 方法：

  • 重写 hashCode() 之后，返回值可能相同，但 identityHashCode() 不会相同：因为 identityHashCode() 返回对象物理地址产生的 hash 值

重写 equals() 后应检查是否符合：

  • 对称性
  • 传递性
  • 一致性
  • 自反性
  • 非空性

New from JDK 1.8

由于从 Java 8 开始引入了函数式编程，因此 Map 也实现了与此相关的新接口：

内部接口 Map.Entry

  • Map 中的 keySet，values，entrySet 与其息息相关
  • 沿用了 Map 接口的范型 <K, V>
  • 代表 Map 中每一个单独的元素

子类及接口

AbstractMap

  • 为 Map 的实现类预先实现了许多 Map 接口声明的方法
  • 但这些方法大多依赖于 entrySet 方法：
    • AbstractMap 并未实现 entrySet 方法，交给子类实现

另：提供了两个实现了 Map.Entry 接口的静态内部类

SortedMap 接口

  • 继承自 Map
  • 接口暴露了用于排序的比较器对象，且定义的方法可以获得集合的子集视图
  • 要求其实现类必须保证内部元素可以按自然或 Comparator 的顺序排列

NavigableMap 接口

  • 包含用于在映射表中查找和遍历的方法
  • TreeMap 实现了该接口

非线程安全组合：

  • HashMap：存储 k-v 关联的数据结构
  • TreeMap：键值有序排列的映射表
  • EnumMap：键值属于枚举类型的映射表
  • LinkedHashMap：可记住 k-v 项添加次序（有序）的映射表
  • WeakHashMap：其值不需要后可被 gc 回收的映射表，一定程度避免内存溢出
  • IdentityHashMap：使用 == 而不是 equals 比较键值的映射表

涉及的基本数据结构：
散列集
散列表（hash table）：为每个对象计算一个整数，称为散列码（hash code）

  • 散列码由对象的实例域产生的一个整数，具有不同数据域的对象产生不同的散列码
  • 如果自定义类，需实现该类的 hashCode() 方法（该方法应与 equals 方法兼容）
    Java 散列表：使用链表数组实现
  • 每个列表被称为“桶”（bucket）
  • 要想查找表中对象的位置，须先计算散列码，与桶的总数取余，所得为保存该元素的桶的索引
  • 有的时候会遇到桶被占用的情况——散列冲突（hash collision）
    • 需要将新对象与桶中的所有对象进行比较，查看该对象是否已经存在
    • 如有相同 Key 的元素：将其 value 更新
    • 否则：将该元素作为新的头元素加入桶中
    • 如散列码设计合理且随机分布，桶的数目足够大，比较次数也会少
  • 如想更多控制散列表运行性能，则需要指定一个初始的桶数：用于收集具有相同散列值的桶的数目
  • 如大致知道最终会有多少个元素要插入到散列表中，便可设置桶数：通常为预计元素个数的75%-150%
  • 若散列表太满：需要再散列（rehashed）
    • 创建一个桶更多的表，并将所有元素插入到新表中，再丢弃原来的表
    • 装填因子（load factor）决定何时对散列表进行再散列
      • 如设为0.75（默认值），而表中超过75%位置已填入元素：用双倍的桶数自动进行再散列

实现：
最简单的 set 类型：没有重复元素的元素集合

  • Java 集合类库提供 HashSet 类：实现基于散列表的集
  • 使用 add 添加元素
  • contains （被重新定义）快速查看某元素是否已在集中（只在某个桶中查找元素，不必查看集合中的所有元素）
  • 散列集迭代器依次访问所有桶。因散列将元素分散在表的各个位置上：访问顺序随机
  • 在不关心集合中元素顺序时才应使用 HashSet

映射表

  • 存放 k-v 对，可通过键查找到值
  • Java 类库提供了两个通用的实现：HashMap 和 TreeMap——均实现了 Map 接口
    • 散列映射表对键进行散列
    • 树映射表用键的整体顺序对元素排序，并将其组织成搜索树
  • 散列或比较函数只能作用于键，与键关联的值不能进行散列或比较
  • 如不需要按照排序访问键，就最好选择散列
    • 如果不能通过 get() 得到，返回的则是 null
  • 键必须唯一
    • 不能对同一个键存放两个值
    • 如对同一个键两次调用 put()，第二个值会取代第一个值（put 将返回用这个键参数存储的上一个值）

注：集合框架并没有将映射表本身视为一个集合

  • 其他数据结构框架将其视为对（pair）集合，或用键作为索引的值的集合
    可以通过已经实现的方法获得映射表的视图——实现了 Collection 接口对象，或者它的子接口的视图

  • 键集
    Set keySet() // 实现Set接口某个其他类的对象

  • 值集合（非集，因为会有重复）
    Collection values()

  • 键/值对（k-v）集
    Set<Map.Entry<K, V>> entrySet()

  • 元素为静态内部 Map.Entry 对象

  • 可通过枚举 entrySet 各个条目来查看键和值

  • 键与键/值对形成一个集：在映射表中一个键只能有一个副本
    注：只添加键不添加值是无意义的。

实现类：

HashMap

TreeMap

  • 实现了 NavigableMap 接口
  • 包含静态类 TreeMap.Entry
  • 数据结构为红黑树，保证元素的有序遍历

HashSet

  • 继承自 AbstractSet，内部有一个 HashMap 成员变量
    private transient HashMap<E,Object> map;
    即：其底层实现其实是由 HashMap 实现

  • HashSet 核心为一只使用 HashMap 的 Key 及 KeySet 的子类

  • 每次往 HashSet 添加元素时，都会将元素作为 Key 添加进来

  • 同时新建 Object 对象用于填充内部 HashMap 的 Value

  • 不允许重复的值

专用集：
弱散列映射表

  • 只要映射表是活动的，则其中的所有桶都是活动的
  • 如某个对象只能被弱引用，则垃圾回收器可以将其回收，但要将引用这个对象的弱引用放入队列中
  • 弱引用进入队列意味着这个键不再被他人使用
  • WeakHashMap 使用弱引用保存键，并周期性检查队列，以便找出新添加的弱引用
  • 保证垃圾回收时 key 能被回收，导致 value 也能被回收，从而使对应的 entry 消失

链接散列表和链接映射表
LinkedHashSet

  • HashSet 的子类
  • 使用双向链表记录插入顺序，遍历时依照该插入顺序
  • 对于同一个元素，其插入顺序不会因多次插入而变化。只有当其被移除后再次插入时，顺序才会改变

LinkedHashMap
当条目插入到表中时，会被并入双向链表中

  • LinkedHashMap 使用访问顺序对映射表条目进行迭代，而非插入顺序
  • 每次调用 get 或 put：受影响（被改动过的）条目将从当前位置删除，并放到条目链表尾部
    • 只有条目在链表中的位置受影响，散列表中的桶的位置不受影响
    • 构造：

访问顺序对于实现高速缓存的“最近最少使用”（LRU）原则十分重要

  • 将访问频繁的元素置于内存，不频繁的放入数据库
  • 在表中找不到元素项且表已满时，可将迭代器加入表中，并将枚举的前几个元素（近期最少使用的元素）删除
    自动化方法：
  • 覆盖 LinkedHashMap 的 protected boolean removeEldestEntry(Map.Entry<K, V> eldest)
  • 每当方法返回 true 时，添加一个新条目，从而删除 eldest 条目
  • 另还可对 eldest 进行评估，以此决定是否应该将它删除

枚举集与映射表
EnumSet：枚举类型元素集的高效实现

  • 底层是位向量（bit vector）
  • 枚举类型序列有限：EnumSet 使用位序列实现；如对应值在 EnumSet 中，则相应的位被置为1
  • 使用静态工厂法构造该集

EnumMap：键类型为枚举类型的映射表

  • 底层是数组
  • 可直接且高效地用一个值数组实现

标识散列映射表
IdentityHashMap：键的散列值使用 System.IdentityHashCode() 计算的

  • 根据对象的内存地址来计算散列码
  • 允许 key 重复
  • 在两个对象进行比较时，IdentityHashMap 使用 == ，不使用 equals()
  • 即：不同的键对象，即是内容相同，也被视为不同的对象

TreeSet

  • AbstractSet 的子类，同时实现 NavigableSet 接口
  • TreeSet 与 TreeMap 的关系，如同 HashSet 与 HashMap 的关系
  • TreeSet 的元素保证有序，按照自然顺序或指定 Comparator 排序规则排序
  • TreeSet 的 descendingSet，headSet 等方法与 TreeMap（NavigableMap）中类似名称的方法起着类似作用
