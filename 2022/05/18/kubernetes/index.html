Kubernetes 概述

Posted on 2022-05-18 Views:

Kubernetes 是 Google 在 2014 年发布的 Omega 开源版本，是全球发展最快、市场占有率最高的容器编排引擎产品。

Kubernetes 源于 Google 开发的 Borg，最初用于调度（包括上线、下线、重启、应用缩扩容等）公司内部近 20 亿个容器，积累了数十年的容器编排经验。

Kubernetes 属于云原生的一部分，相当于容器的“操作系统”。云原生其它内容还包括 ConfigMap（配置中心）、etcd（发现中心，负责存储和通知）、service mesh（网格服务，sidecar，为每个服务提供网关、鉴权等的载体）、CI/CD。

主要构成

既然是基于容器技术的分布式集群管理系统，Kubernetes 也应该有主（Master）从（Node）的概念。
除此之外还包括 Kubernetes 的指令系统（kubectl）和存储管理（etcd）等。

简图如下：

  • Master 统一管理 Node 节点，在工作程序运行有效负载（应用程序）的地方控制和监视集群
  • Node 在分布式系统中可以是物理主机，也可以是虚拟机
  • Node 节点和 Master 节点共同组成分布式集群的 Cluster

一个 Kubernetes 集群最好能拥有 3 个 Master 节点，以实现高可用（HA Cluster）。

Master

Master 各组件如图：

kube-apiserver 进程

为一 RESTful 服务器，是集群内各个功能模块之间数据交互和通信的中心枢纽

  • 提供了集群管理的 API 接口，也即 Kubernetes 的统一接口
  • 提供了完备的集群安全机制（验证，授权）
  • 同时也对集群外部暴露服务
  • 用户可通过 REST 接口或 kubectl 命令行工具进行集群管理；因此 kubectl 本质上也是与 kube-apiserver 进行通信。

Master 和 Node 中几乎所有的组件都需要和 apiserver 通信，以执行所需要的职责。
这样模块之间不会直接互相调用，而是统一通过 apiserver 打交道完成通信；并且屏蔽了与 etcd 的直接交互。

kube-scheduler 调度程序

调度程序通过执行一系列复杂的算法（比如轮询调度 RR），为每个 pod 匹配到一个最佳的目标 node，然后分发任务。

kube-controller-manager

守护进程。顾名思义，是 Kubernetes 管理各个 service controller 的控制中心。

每个 controller 本质上是一个控制循环，它通过 apiserver 监视集群的共享状态，对服务进行监控和恢复，尝试改变某个服务与期望不符的实际情况。

Kubernetes 内部的 service controller 包括：

  • Cronjob Controller
  • DaemonSet Controller：能让特定或所有节点运行同一个 pod
    • 节点被加入 Kubernetes 集群后：pod 会被 DaemonSet 调度到该节点执行
    • 节点被移出 Kubernetes 集群后：被 DaemonSet 调度的 pod 会被移除
    • 删除 DaemonSet：所有与其相关的 pod 都会被移除
  • Deployment Controller（包括了 Replication Controller 的所有功能）
  • Endpoint Controller
  • HPA(Horizontal Pod Autoscaling) Controller
  • Job Controller
  • Namespace Controller
  • NodeLifecycle Controller
  • PV-binder Controller
  • PV-expander Controller
  • Replication Controller：通过 ReplicaSet 控制 Pod 的动态扩容和缩放
    • 为每个 Pod 设置一个期望的副本数
    • 需要时复制多份 Pod，不需要后缩减 Pod 至指定份数
    • 期望值可手动更新，或自动扩容代理完成
  • ResourceQuota Controller
  • ServiceAccount Controller
  • Service Controller：管理 service
  • StatefulSet Controller
  • Token Controller：对 API server 访问进行认证

etcd 组件：作为高可用强一致性的存储仓库，存储集群上包括 Kubernetes 对象，node, pod, service, replicaset, namespace 等所有集群配置信息。

etcd 组件可内置于 Kubernetes，也可于集群外部搭建供 Kubernetes 使用。

etcd 的增删改查也是统一通过 kube-apiserver 进行调用的。

Node

node 包含以下状态信息：

  • 地址：地址组合用法取决于云服务商或者裸金属配置
  • 状态：OutOfDisk, Ready, MemoryPressure, DiskPressure
  • 容量：node 上的可用资源：CPU、内存和可以调度到 node 上的 pods 的最大数量
  • 信息：通用信息——内核版本、Kubernetes 版本（kubelet, kube-proxy 版本），容器版本，操作系统名
Node 状态 描述
OutOfDisk True：node 的空闲空间不足以添加新 pod
Ready True：node 是健康的，并已经准备好接收 pod
False：node 不健康，且不能接收 pod
Unknown：node 控制器在最近 40 秒内没有接收到 node 的消息
MemoryPressure True：node 不存在内存压力：内存用量低
DiskPressure True：node 不存在磁盘压力：磁盘用量低

kubelet 组件

  • Node 的心脏，每个 Node 节点运行一个 kubelet 进程
  • 与 apiserver 通信，并运行为其节点安排的容器
  • 管理容器生命周期（创建、修改、监控、删除等）
  • 结合 cAdvisor 负责向 Master 汇报自身节点运行情况：Node 节点注册、终止、定时上报健康状况等
  • 接收 Master 发出的命令，创建相应的 Pod

kube-proxy：

  • 使用 IP 表（iptables）/IPVS 处理 Pod 的网络需求
  • 利用 service 负责服务地址到 Pod 地址的代理，负载均衡等
  • 监听 apiserver 中 service/endpoints 变化，为每个 service 建立服务代理对象（VIP），并刷新负载均衡
  • 实现了集群内部从 pod 到 service，以及集群外部从 nodePort 到 service 的访问

kube-proxy 的实现模式：

userspace：最早的负载均衡方案（default before Kubernetes v1.1）

  1. 访问服务的请求到达 node 后，先进入内核的 iptables
  2. 请求被转到给用户空间监听的 kube-proxy 端口
  3. kube-proxy 处理后，将请求封装
  4. kube-proxy 再根据内核空间的 iptables 中的规则，将请求分发给后端的 service(backend) pod

流量从用户空间进出内核带来的性能损耗是不可接受的

iptables：目前的默认代理模式
基于 netfilter 实现

  • 当客户端发起请求时，直接请求本地内核的 service IP，根据 iptables 规则直接路由到各 pod 上
  • kube-proxy 负责维护 node 上的防火墙规则和路由规则，根据 service / endpoints 对象改变实时刷新规则
  • iptables 使用 NAT 完成转发，采用随机数实现负载均衡
    与 userspace 模式最大区别：
  • iptables 使用 NAT 模块实现 service 入口地址到 pod 实际地址的转换，免去一次内核态到用户态的切换
  • 如 iptables 代理最初选择的那个 pod 没有响应：则不会自动重试其他 pod
    性能问题：
  • 使用 NAT 转发，存在不可忽视的性能损耗
  • 如集群中存在上万的 service / endpoint，则 node 上的 iptables rules 将非常庞大

ipvs：since Kubernetes v1.9-alpha, default since v1.11

  • 客户端请求到达内核空间时，根据 ipvs 规则直接分发到各 pod 上
  • kube-proxy 监视 Kubernetes 的 service / endpoints 对象，调用 netlink 接口，相应地创建 ipvs 规则
  • kube-proxy 定期与 Kubernetes 地 service /endpoints 对象同步 ipvs 规则，以确保 ipvs 状态与期望的一致
  • 访问服务时，流量将被重定向至其中一个后端 pod
    ipvs（基于哈希表）的性能优于 iptables（基于链表）
    ipvs 支持的轮询算法：
  • DR: Direct Routing
  • NAT
  • Tunneling，也称 ipip 模式
  • rr: round-robin/轮询
  • lc: least connection/最少连接
  • dh: destination hashing/目标哈希
  • sh: source hashing/源哈希
  • sed: shortest expected delay/预计延迟时间最短
  • nq: never queue/从不排队

Pod：最基本的操作单元

可根据标签区分 pod 要运行在哪台主机上

Kubernetes 使用三种资源管理存储

  • PV
  • Persistent Volume Claim（PVC）
    • namespace 里的资源，描述一个用户对 PV 的请求
    • 请求信息包含存储大小、访问模式等
  • StorageClass
    • 为 PV 创建模板，为管理员提供一种描述存储“类型”的方法
    • 不同类型可映射到服务质量级别、备份策略、或群管理员确定的任意策略的 PV 上
    • 在其他存储系统上会被称为 profiles

Persistent Volume（持久化卷，PV）

  • 对底层共享存储的抽象
  • 由 admin（管理员）进行创建和配置
  • 通过插件机制完成与共享存储的对接
  • 其生命周期与使用它的 pod 相互独立

应用场景：
用户提交 PVC，在 PVC 中指定存储类型（storageClassName），如果符合已经定义好的 StorageClass，则会为其自动创建 PV 并进行绑定。

容器编排

Pod

Service

service 解决的问题：

  • 每个 pod 都有一个 ip 地址：地址不稳定，随着 pod 销毁和创建发生变化
    • 利用非固定 ip 进行网络调用不现实
  • 服务调用需要对不同 pod 进行负载均衡

Kubernetes 抽象一组 pod 的逻辑定义和访问它们的方法——service

  • An abstract way to expose an application running on a set of Pods as a network service.
  • 充当代理进行负载均衡

工作原理：

  • 每个 pod 会设置标签 label
    • 实际运用时会打上类似 “app=xxx” 的标签
  • service controller 通过 serviceSelector 选择相同 label 的 pod
    • 按上例：创建 “app=xxx” 的 serviceSelector，为 service 服务
    • 列举并构建出一个 endpoints：负载均衡列表，可被包装成完整的 service，对外提供服务
  • 每个 service 有固定的 cluster ip（service 对象的 ip, VIP）和一些 endpoints
    • kube-proxy 通过 iptables / ipvs 为 service 实现 VIP
    • endpoints 所有 ip 和 pod 的端口通过 service controller 和 endpoint controller 共同选择配置完成
  • endpoints 设置在 pod 上，确定标签选择器
    • 如果 pod 出现故障：自动在 endpoints 被移除
    • 如 serviceSelector 匹配到新的 pod：自动加入各自对应的 endpoints 对象中
  • service 职责：与每个 node 的 kube-proxy 进程相结合，将“数据包”分发到相应节点时充当负载均衡
    • kube-proxy 负责实现 service 的 VIP
    • 某些类型的 service 必须要与 kube-proxy 相结合
  • 将服务信息通过 apiserver 存入 etcd 中

例：

端口解析：
port

  • 表示 service 暴露在 clusterIP 上的接口，clusterIP:Port 是提供给集群内部访问 Kubernetes 服务的入口
    targetPort
  • containerPort，targetPort 是 pod 上的端口
  • 从 port 和 nodePort 上到来的数据经过 kube-proxy 流入到后端 pod 的 targetPort 上进入容器
    nodePort
  • nodeIP:nodePort 为提供给从集群外部访问 Kubernetes 服务的入口

即：port 和 nodePort 都是 service 的端口，前者暴露给从集群内访问服务的方式，后者则暴露给集群外访问服务的方式
这两个端口的流入数据均需经过反向代理 kube-proxy 流入后端具体 pod 的 targetPort，从而进入到 pod 上的 container 内

IP 解析：
Pod IP

  • Pod 的 IP
  • 每个 pod 启动时，内部会启动一个 pause 容器（Google 的一个镜像：gcr.io/google_containers/pause）
  • Pod 内部其他容器的网络模式使用 container 模式，并指定为 pause 容器的 ID（network_mode: “container:{pauseContainerID}”）
    • 使得 pod 内所有容器共享 pause 容器的网络，与外部的通信经由此容器代理
  • pause 容器的 IP 也可称为 Pod IP
    Cluster IP
  • Pod IP 地址实际存在于某个网卡（或虚拟设备）上
  • Cluster IP 并没有网络设备承载这个地址：其为一个虚拟地址
    • service 对象的 ip
  • 由 kube-proxy 使用 iptables 规则重定向到其本地端口，再均衡到后端的 pod
  • 当 kube-proxy 发现一个新的 service 后，会在本地节点打开一个任意端口，创建相对应的 iptables 规则
  • 重定向服务的 clusterIP 和 port 到该新建的端口，开始接受到达该服务的连接
    Node IP
  • 为集群外的客户提供业务，为该服务提供公共 IP
  • 指定 service 类型为 NodePort 后，系统会给它在集群的各个代理节点上分配一个节点级别的端口
    • 能访问到代理节点的客户端都能访问该端口，从而访问到服务

Service 的不同类型（type）：
ClusterIP

  • 默认的 Service 类型
  • 根据是否生成 ClusterIP 又可分为：
    • 普通 service：通过为 Kubernetes 的 service 分配一个集群内部可访问的固定虚拟 IP（Cluster IP），实现集群内的访问，最常见
    • Headless Service
  • 只用于集群内部通信，服务间只需访问 VIP 即可负载均衡地访问到 service 中的 pod
    NodePort
  • 除了使用 Cluster IP 外，还通过将 service 的 port 映射到集群内某几个节点（node）的相同一个端口，实现通过 nodeIP:nodePort 从集群外访问服务
    • 即：通过 iptables，使得调用 node 上的该特定 port 就能访问其背后的 service
  • kubernetes 控制台从默认 30000-32767 中分配一个端口来代理服务
  • 或：在 .spec.ports[*].nodePort 配置指定的端口值
  • 亦可调整 kube-proxy 中的 –nodeport-addresses 来配置指定的代理 IP（since kubernetes v1.10）

LoadBalancer

  • 与 NodePort 类似
  • 除了使用一个 Cluster IP 和 NodePort 外，还通过所使用的公有云申请一个负载均衡器，实现从集群外通过 LoadBalancer 访问服务
    • 负载均衡器后端映射到各节点的 NodePort
    • 流量直接到后端的 pod 中
      ExternalName
  • Service 的特例，主要面向集群外部的服务
  • 通过 ExternalName 可将外部服务映射进 Kubernetes 集群，且具备 Kubernetes 内服务的一些特征（如 namespace 等），为集群提供服务
  • kube-dns version >= 1.7
  • 该重定向依赖于 DNS，不像上述三种类型（通过 kube-proxy）

备注：

  • 前三种类型通过 selector 指定服务对应的 pod，根据 pod 地址创建出 endpoints 对象作为服务后端
    • Endpoint Controller 监测 service 和 pod 的状态，维护对应 endpoints 信息
    • kube-proxy 根据 service 和 endpoint 维护本地路由规则
    • pod 发生变化 -> service 发生变化 -> endpoints 发生变化 -> kube-proxy 在每个 node 上更新 iptables，实现一层负载均衡
  • ExternalName 不通过 kube-proxy 重定向：不指定 selector， 故没有 port 和 endpoint
  • ExternalName 与 ClusterIP 的 Headless Service：同属 Headless Service
    • 不分配 service IP，且不通过 kube-proxy 做反向代理和负载均衡

服务发现：
环境变量

  • pod 创建完成后，kubelet 会在该 pod 中注册该集群已创建的所有 service 相关的环境变量
  • 注意：service 创建之前的所有 pod 并不会注册该环境变量
    DNS
  • 默认形式
  • 可在集群中部署 CoreDNS 服务（旧版本 Kubernetes 使用的是 kubeDNS），达到集群内部的 pod 通过 DNS 方式进行集群内部各个服务之间的通信
